# State Delta Tombstone Replay Fix

Last updated: 2026-06-22

This note is a technical handoff for fixing the legacy Koinos node `verify-blocks = false` replay path.
It documents the mainnet mismatch reproduced locally in Koinos One/Teleno and the minimum behavior required
for protocol-compatible stored-delta replay.

## Summary

Fast block replay from stored state deltas must preserve remove entries as delta tombstones, even when the
removed object is absent from the parent state.

The current fast replay behavior can treat an absent-key remove as a no-op. That is incorrect for historical
state delta Merkle roots. A block can legitimately end with a remove tombstone for an object that is not present
in the parent state, for example when execution creates and removes the object inside the same block. The visible
post-block key-value state is unchanged for that key, but the block delta root is different.

This is enough to make a restarted node using `verify-blocks = false` diverge and later fail with:

```text
block previous state merkle mismatch
```

The database does not need historical delta rewriting. The replay semantics need to match full validation.

## Important Finding: Not A Transaction-Delta Problem

The initial suspicion was that fast replay should also apply `transaction_receipts[].state_delta_entries`.
That is not the correct fix.

For block `1,628,526`:

```text
block_receipt.state_delta_entries: 19
transaction_receipts[].state_delta_entries: 18
exact block/transaction overlap: 17
```

Tested replay strategies:

```text
block receipt deltas only: no match
transaction receipt deltas only: no match
transaction then block: no match
block then transaction: no match
prefixes/subsets: no match
```

Full validation regenerated byte-identical block and transaction deltas compared with the stored receipt:

```text
block delta hash: c71e96adf75acfd06a9ca23cd551a18a6ab665e5bc1e0793c7bf75122209dc8f
transaction delta hash: 749660af41b81e192bb65b598b5c133c6515a4debb972fedbaf03561f2d81355
```

So the stored final block deltas are the right source. The replay behavior for remove entries is wrong.

## Root Cause

The problematic block receipt includes two remove entries. One of them removes an object that is absent from
the parent state:

```text
block receipt delta index: 9
action: remove
object_space.system: false
object_space.id: 5
object_space.zone: 0x0086899d9be6e468ed1333f488772920c016da3c56daa1c5c4
key: 0x
parent_has_object: false
```

In the Koinos One/Teleno code path, fast replay maps a receipt remove to `remove_object()`:

```cpp
if( delta_entry.has_value() )
  block_node->put_object( object_space, delta_entry.key(), &delta_entry.value() );
else
  block_node->remove_object( object_space, delta_entry.key() );
```

The state delta implementation records a remove only when the key exists:

```cpp
void state_delta::erase( const key_type& k )
{
  if( find( k ) )
  {
    _backend->erase( k );
    _removed_objects.insert( k );
  }
}
```

That is valid for normal user-facing state mutation APIs. It is not sufficient for replaying a serialized
historical delta, because a serialized remove entry is already part of the historical delta shape. Dropping it
changes the delta Merkle root.

## Simple Example

Parent state:

```text
A = 10
B = 20
C absent
```

During full block execution:

```text
put C = 30
remove C
```

Visible state after the block:

```text
A = 10
B = 20
C absent
```

Serialized final block delta may include:

```text
remove C
```

Fast replay cannot decide that `remove C` is a no-op just because `C` is absent in the parent. The remove entry
is a tombstone and must be preserved in the replay delta so the Merkle root matches historical full validation.

## Recommended Legacy Fix

Do not change normal `remove_object()` semantics globally. Normal state mutation should still be free to treat
absent-key removal as a no-op if that is the established API behavior.

Add an explicit replay-only path that preserves serialized remove tombstones.

Preferred shape:

```cpp
// State DB API used only by stored-delta replay.
void remove_object_from_delta_replay(
  const object_space& space,
  const object_key& key,
  bool preserve_tombstone );
```

or:

```cpp
void remove_object(
  const object_space& space,
  const object_key& key,
  remove_mode mode );

enum class remove_mode
{
  normal,
  preserve_tombstone
};
```

Then in `apply_block_delta()`:

```cpp
for( const auto& delta_entry: receipt.state_delta_entries() )
{
  auto object_space = convert_object_space( delta_entry.object_space() );

  if( delta_entry.has_value() )
  {
    block_node->put_object( object_space, delta_entry.key(), &delta_entry.value() );
  }
  else
  {
    block_node->remove_object( object_space,
                               delta_entry.key(),
                               remove_mode::preserve_tombstone );
  }
}
```

The replay-only remove must insert the key into the delta's removed-object/tombstone set even when lookup in the
parent state returns absent.

Avoid this as the final production API if possible:

```cpp
// Works as a proof but is not the clean production interface.
if( !node->get_object( space, key ) )
  node->put_object( space, key, &empty_value );
node->remove_object( space, key );
```

That workaround reproduced the correct root in the audit tool, but it encodes replay semantics indirectly.
An explicit tombstone-preserving remove is clearer and less fragile.

## Required Safety Check

After applying any stored block delta, fast replay should verify the computed post-state root.

Use, in order:

1. `receipt.state_merkle_root`, when present.
2. The next block header's `previous_state_merkle_root`, when the receipt root is missing and the next block is
   available.

If the computed root does not match, discard the tentative replay node and fall back to full block validation
for that block. This prevents silent divergence on any other historical edge case.

## Notes For Operators

This is a replay bug, not evidence that the historical mainnet block deltas are corrupt.

Do not fix this by rewriting historical block records or state deltas. The stored block deltas matched full
validation byte-for-byte for the reproduced block. The replay path must preserve the semantics encoded in those
deltas.

For recovery on an already affected node, the conservative workaround remains:

```text
restart as observer with verify-blocks=true
let full validation repair/advance the local state
then only use verify-blocks=false after the replay fix is installed
```
